# BidRadar-X DOCX 报告格式

版本：TASK-05 / 2026-07-14

## 1. 边界与入口

独立生成器位于 `backend/app/services/docx_publisher.py`。它不读取工作流状态，不判断增量，也不调用现有 JSON Publisher；调用方必须先准备经过 `docs/DATA_CONTRACT.md` 校验的 `TenderNotice` 列表。

公开入口：

```python
path = DocxPublisher(output_dir=report_dir).publish(
    query=user_query,
    notices=validated_notices,
    report_scope="full",  # 或 "incremental"
)
```

`publish` 返回生成文件的 `Path`。生成成功前会重新打开文件并验证段落、表格、必需内容和外部超链接；验证失败时删除无效工件并抛出异常。

## 2. 报告范围

| `report_scope` | Word 中的名称 | 输入列表语义 |
| --- | --- | --- |
| `full` | 全量报告 | 调用方传入本次应交付的全部公告 |
| `incremental` | 仅新增内容报告 | 调用方只传入已由增量模块确认的新公告 |

生成器只忠实呈现输入列表，不自行猜测“新增”、查询历史快照或补造变化。实质变化报告仍应由后续增量任务按统一数据契约整理为明确的交付输入。

## 3. 文件名与冲突

文件名固定为：

```text
{Windows安全的用户问题}_{YYYYMMDDHHmm}.docx
```

- 保留中文和普通空格。
- 将 Windows 禁止字符 `< > : " / \ | ? *`、控制字符替换为下划线，合并连续下划线，并清理结尾空格和句点。
- 空名称使用“情报任务”；`CON`、`PRN`、`AUX`、`NUL`、`COM1`—`COM9`、`LPT1`—`LPT9` 等保留名加前缀下划线。
- 用户问题部分最多保留 80 个字符。
- 同一输出目录、同一用户问题、同一分钟的目标文件已存在时，抛出 `FileExistsError`；不覆盖、不改写原文件，也不静默生成含糊的副本名。
- 默认生成时间使用 `Asia/Shanghai` 时区。

## 4. 文档结构

### 4.1 报告头

报告标题明确标注“全量报告”或“仅新增内容报告”。其后为元数据表：

1. 用户问题；
2. 报告范围；
3. 带时区的生成时间；
4. 公告数量。

无结果时只写“未发现符合条件的公告”，不创建虚构公告或附件。

### 4.2 公告顺序和必需内容

公告按 `published_at` 降序排列，相同时按标题降序排列。公告之间分页。每条公告包含一个五行事实表：

| 字段 | 内容规则 |
| --- | --- |
| 标题 | `TenderNotice.title` 原值 |
| 发布时间 | `published_at`，保留分钟和时区 |
| 来源链接 | 显示来源名称，并提供可点击的 `source.source_url` |
| 核心内容 | `core_content` 原值，不用模板摘要替代 |
| 附件链接 | 每个 `attachments[].url` 都可点击；有名称时显示名称，无名称时显示 URL；空列表严格显示“无” |

### 4.3 客观要求与证据

`requirement_sections` 按输入顺序呈现：

- 章节标题和摘要只呈现契约中的现有值。
- 事实表包含“字段、内容、证据”。已知值显示原值；未知值显示“未知（`unknown_reason`）”。
- 可变宽表格保留原列，并附加“证据”列；空单元格显示“未知”。
- 每个 `evidence_id` 输出可点击的 `source_url`，并呈现实际存在的文档名、页码、章节/定位符和原文片段。
- 不从旧固定模板补齐缺失章节、字段、附件、证据或推断性内容。

## 5. 回读验证

`validate_docx` 使用 Word 文档模型重新读取落盘文件，并至少检查：

- 文档可打开且包含段落和表格；
- 报告范围标签存在；
- 每条预期公告都有结构正确的五行事实表；
- 标题、发布时间、来源名称和核心内容存在；
- 无附件公告的附件单元格精确为“无”；
- 每个来源及附件 URL 都由 DOCX 外部超链接关系引用，且目标为 HTTP(S)。

验证结果返回段落数、表格数和超链接目标列表，供测试或后续下载层审计。生成器本身不接入 API、前端或总工作流。
