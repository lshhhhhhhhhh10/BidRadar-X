# BidRadar-X Word 报告格式与事实约束

更新时间：2026-07-19（Asia/Shanghai）

生成器位于 `backend/app/services/docx_publisher.py`。系统为每个有效项目生成独立 DOCX；没有项目时不生成空文档。

## 文件名

```text
{用户问题}_{YYYYMMDDHHmm}.docx
```

文件名会清理 Windows 禁止字符、控制字符、保留设备名和过长文本。同名冲突不会覆盖已有文件。

## 文档信息架构

### 1. 商务简报页眉

- BidRadar-X 招投标机会研判标识；
- 报告日期与页码；
- 检索问题、范围、项目数量、生成时间和事实基础；
- 报告口径说明：事实由程序锁定，AI 只做辅助研判。

### 2. 一页速览

- 项目数量；
- 最近截止时间；
- 已披露预算；
- 证据来源数量；
- 项目清单与风险分布。

多币种预算不会被错误直接相加；来源未披露预算时明确显示未披露。

### 3. AI 辅助摘要与风险研判

包含：

- 执行摘要；
- 风险判断；
- 可关注机会；
- 建议行动。

模型输出必须引用输入证据 ID。AI 不得修改标题、发布时间、采购人、来源 URL、项目编号或原文事实。模型不可用时报告保留事实内容，并写入真实失败原因，不把规则文案伪装成 AI 结论。

### 4. 单项目事实

- 项目标题；
- 发布日期、地区、来源；
- 项目编号、采购人、预算、截止时间；
- 联系人、电话和代理机构；
- 原公告核心内容；
- 原公告和附件链接。

### 5. 招标条款与证据

根据公告实际披露内容展示预算、截止时间、资格资质、保证金、工期、评审办法、递交地点和付款结算等指标。未在原文或已下载文件中找到的指标默认不展示；用户选择全部指标时明确显示“未找到可核验原文”。

每条证据尽量保留：

- 来源名称与 URL；
- 文档名；
- 页码或章节定位；
- 原文片段；
- 证据 ID。

## DOCX 回读验证

落盘后系统重新打开文档并检查：

- 文档可读取且具有段落/表格；
- 每个预期项目均存在；
- 标题、发布时间、来源和核心内容完整；
- 原公告和附件的 HTTP(S) 超链接关系存在；
- 无附件时明确标注，不生成虚假下载链接。

验证失败时不会把文件登记为成功报告。

## 事实一致性原则

1. 原文事实不交给 AI 改写。
2. AI 只能引用本次输入中的证据 ID。
3. 缺失事实显示未知，不根据常识补齐。
4. AI 摘要与风险研判是辅助信息，不替代招标文件和人工法律审查。
5. 报告中的来源链接和附件链接用于回到原始证据。
