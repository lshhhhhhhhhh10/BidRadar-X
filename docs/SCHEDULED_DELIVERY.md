# 定时推送与可靠性

## 用户链路

1. 用户在检索框中输入“每日”或“每周”任务并开始检索。
2. 系统先立即执行一次真实检索。
3. 首次检索完成后，系统以同一 `task_id` 自动建立持久化订阅，不会重复建立相同任务。
4. 如果用户写了“每天 9:30”或“每周三 14:00”，按该时间执行；只写频率时，每日默认 09:00，每周默认周一 09:00，时区为 `Asia/Shanghai`。
5. 新结果继续进入原有的去重、证据审核和单项目 Word 文档生成链路，已推送过的内容由水位线防止重复交付。

## 已实现的可靠性保障

- 订阅、下次执行时间、重试次数和错误都保存在 SQLite，刷新页面不会丢失。
- 执行前用数据库租约原子抢占任务，两个工作器不会同时执行同一个任务。
- 执行中持续续租；工作器崩溃后租约到期，其他工作器可接管。
- 失败后按退避时间重试，默认最多 3 次；最终失败保留可查错误。
- 后端重启时会重新扫描已到期任务，不依赖浏览器页面常开。

## 本地运行的边界

定时工作器在后端进程中运行，因此“关闭浏览器”不影响任务，但“关机或停止后端”时无法在当时发起网络请求。恢复后系统会补跑。如果要求关机时仍准时执行，应将后端和 SQLite 迁移到一台 24 小时在线的服务器，并由进程管理器设置异常自动重启。

## 发布前检查

- 后端健康检查可访问。
- SQLite 数据目录可写且纳入备份。
- 服务器时区与 NTP 时钟正常。
- 用一个两分钟后到期的测试订阅验证“执行、落库、文档交付、下次时间”四步。
