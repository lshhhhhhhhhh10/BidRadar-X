# 事实一致性与测试隔离

## 生产事实闸门

报告生成前的 `deterministic_evidence_binding_v1` 会对每个候选项目执行以下确定性检查：

1. 每个 RAG 证据 ID 必须存在，并且归属于同一项目。
2. 摘要必须是可以在已采集正文或已解析附件中定位的片段。
3. 预算、截止时间、采购人等结构化字段必须同时满足“值一致”与“有字段级证据”。
4. 来源引文必须能在对应公告正文或附件文本中定位。
5. 任一硬性检查失败，该运行不能通过事实核验节点。

模型只负责语义复核，不能修改原始事实字段。报告中的 AI 摘要与风险研判保留证据 ID，与公告原文分区呈现。

## 可复现开发基准

`backend/tests/test_fact_consistency.py` 是小型人工标注开发集，同时包含“可支持事实”和“虚构摘要”。测试会计算 `supported_claims / checked_claims` 并要求虚构事实被拒绝。

这是工程闸门，不是对真实世界“100% 一致”的承诺。赛题验收时应另外抽取不参与开发的公告，由两名标注者独立判定后计算事实精确率、证据覆盖率和标注者一致性。

## 测试数据隔离

`backend/app/storage/database.py` 在检测到 `unittest` 或 `pytest` 进程时，会强制使用按进程分配的临时数据目录，不再读写本地产品库。历史上已写入产品库的 `public-a` / `public-b` / `login-source` 运行已加入隔离表，不会再出现于检索历史与项目报告。
