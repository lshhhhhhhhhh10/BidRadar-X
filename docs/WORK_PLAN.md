本文件负责未来执行顺序和依赖；[ROADMAP](ROADMAP.md) 负责原始构想、旧任务、新能力与当前证据的完整映射；新队员从 [TEAM_HANDOFF](TEAM_HANDOFF.md) 开始。

## 1. 正式编号规则

- **C**：项目上下文、协作和治理。
- **F**：基础契约、存储和可迁移数据底座。
- **R**：真实来源、附件和文档解析。
- **N**：字段规范化、清洗和证据抽取。
- **D**：跨站归并、镜像去重和公告生命周期。
- **I**：增量、变化检测和调度。
- **W**：Word 报告、下载和历史。
- **Q**：端到端、稳定性、安全和比赛验收。
- **L**：核心数据链完成后的高级产品能力。
- **M**：演示、设计文档、操作文档和比赛材料。

`TASK-01`～`TASK-10` 只作为历史执行编号保留；原规划 `TASK-11`～`TASK-14` 已映射到 L/Q/M 能力，不再继续创造 `TASK-15` 等含义不明确的编号。旧任务与新能力是多对多关系，入口见 [ROADMAP 的映射表](ROADMAP.md#旧执行任务到新能力的映射)。

## 2. 状态规则

只使用以下状态：

- `已完成－真实黑盒验证`
- `已完成－自动测试验证`
- `部分完成`
- `原型或实验`
- `外部条件阻塞`
- `未开始`
- `状态无法验证`

“已完成”要求同时有生产实现、测试、工作日志、明确用户或模块入口，没有把 fixture 冒充真实运行，且已知缺口不否定该能力的核心目标。

## 3. 当前唯一关键路径

**下一能力入口仍是 R01：CCGP 公开来源生产加固。**

原因：F01 已定义正式来源契约，F02 已提供迁移、溯源和可靠存储底座；但当前 CCGP 采集器尚未满足 F01 的全局限速、重试/缓存、完整字段与证据、运行状态持久化等生产门禁。R02、R03、R04、N01、D01 和 Q01 都依赖先把 R01 做稳。

C01/C02 只记录治理与交接，不生成 R01 实现提示词，也不开始 R01。

## 5. 推荐执行顺序

1. `R01`：把已有 CCGP 样板提升到 F01/F02 规定的生产契约。
2. `R02 → R03`：取得真实附件并形成 HTML/PDF/OCR 可追溯解析链。
3. `R04`：把 GGZY 升级为第二个可验证公开来源；R05 仅在授权满足后推进。
4. `N01 → N02 → N03 → D01`：统一字段、证据、决策信息和跨站生命周期。
5. `I02 → W04 → Q01 → Q02`：完成变化、产品体验、端到端与稳定性验收。
6. 核心数据链稳定后并行规划 `L01/L04/L05/L06` 与 `L02/L03`。
7. 最后以 `M01` 收口比赛材料；材料可以提前积累，但不能提前标记完成。

## 6. 协作门禁

- 一个能力一个分支、一个 Codex 任务、一个独立工作日志；冲突区域使用 Git worktree 或串行。
- 开始前运行基线；实现任务先写并真实执行红灯测试，再改生产代码。
- 完成后执行全量回归、黑盒、失败路径和敏感信息扫描。
- 只暂存本任务文件，push 功能分支并创建 Draft PR；不直接 push 或自动合并 `main`。
- GitHub 具体步骤见 [GITHUB_WORKFLOW](GITHUB_WORKFLOW.md)。
