# BidRadar-X 项目上下文

更新时间：2026-07-15（Asia/Shanghai）

## 一句话目标

BidRadar-X 把分散、变化快且格式不统一的真实招投标公告转成可追溯、可持续更新、可解释的投标决策与招标辅助信息，并输出可下载的 DOCX 报告。

## 两类用户

### 投标方

用户在聊天框中描述项目主题、地区、时间、企业能力和执行频率。系统查询真实来源，保留历史查询和定时项目，在公告发生实质变化时提示，并展示项目、证据、决策字段、报告时间线与 Word 下载。企业信息向导和企业知识库将用于判断企业能力、资质覆盖或等价关系，而不是只做字符串匹配。

### 招标方

系统未来根据有证据的历史相似中标项目给出预算区间、置信度和依据，并发现潜在供应商及推荐理由。没有足够数据时必须明确不确定性，不能伪造项目、供应商或精确估值。

## 比赛交付目标

- 可重复演示的真实链路产品。
- 符合比赛要求的 DOCX 报告。
- 设计、架构、操作、部署与失败恢复文档。
- 报名材料和演示稿或视频素材。

## 当前事实边界

仓库已经有来源适配器、FastAPI、工作流、SQLite 存储和迁移、调度器、DOCX 发布器以及前端项目/报告链路，但它们的完成度不同。CCGP 有过真实网络黑盒记录；GGZY 当前只有 fixture 自动测试和真实网络失败记录；天眼查招投标 API 已完成 Token 适配器，等待用户凭据进行真实验收。附件、PDF、OCR、企业画像、企业知识库 RAG、资格等价判断、预算估计和供应商推荐尚未达到产品能力。

“当前 RAG”仅指公告字段的本地词法/字符串相似检索样板；它没有接入企业内部资料或飞书资料，也没有文档索引、向量检索和企业知识库问答，因此不能称为企业知识库 RAG。

完整的历史映射、状态证据和缺口以 [ROADMAP](ROADMAP.md) 为唯一总表；执行顺序以 [WORK_PLAN](WORK_PLAN.md) 为准；新队员从 [TEAM_HANDOFF](TEAM_HANDOFF.md) 开始。

## 事实记录规则

1. 生产实现、测试、工作日志和用户/模块入口必须互相核对。
2. fixture 通过不等于真实网络可用；离线解析通过不等于在线授权成功。
3. 本地提交不等于 GitHub 已同步；必须比较本地与远端分支 SHA。
4. 未实现能力使用“部分完成”“原型或实验”“外部条件阻塞”“未开始”或“状态无法验证”，不使用模糊百分比。
5. 旧 `TASK-*` 只用于追溯历史执行，新工作统一使用能力编号。
