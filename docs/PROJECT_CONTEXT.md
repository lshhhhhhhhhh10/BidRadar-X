# BidRadar-X 项目上下文

更新时间：2026-07-14（Asia/Shanghai）

## 1. 文档目的与事实来源

本文档用于统一比赛阶段的产品边界、实现现状和协作约束，避免把模拟能力当成已交付能力。

当前事实来源按优先级排列：

1. 本次项目协调指令中的硬性要求。
2. `README.md`、`package.json`、`backend/`、`app/`、`design-qa.md` 的当前实现。
3. Google Docs《开发日志》（读取于 2026-07-14）。

如后续取得正式比赛规则原文，应逐条补充规则编号、原文和验收证据；在此之前，不得自行扩张比赛范围。

## 2. 比赛硬性要求

### 2.1 当前第一目标

唯一的第一优先级交付链路是：

> 真实数据源 → 清洗去重 → 增量判断 → DOCX

四个环节必须端到端连通，不能以模拟数据、固定模板、结构化 JSON 或禁用的下载按钮替代最终验收。

最低完成定义：

- 真实数据源：至少一个可合法访问的权威招投标来源能够稳定采集真实公告及其必要附件，并保留来源 URL、发布时间和采集时间。
- 清洗去重：真实网页/附件被转换为统一结构；同项目跨站镜像被合并，但原始来源和公告生命周期事件仍可追溯。
- 增量判断：同一任务重复运行时，只把新项目或实质字段变化判定为增量；重复运行不得重复交付相同事件。
- DOCX：生成可正常打开、内容完整、中文显示正确、包含项目事实与来源证据的 `.docx` 文件，并可由产品界面下载。

### 2.2 当前阶段不得扩张的范围

- 企业画像属于后续阶段。
- 甲方预算估计属于后续阶段。
- 供应商推荐属于后续阶段。
- 甲方侧招标编制工具不属于当前第一目标。
- 当前阶段只呈现可由来源证据支持的客观事实，不输出中标概率、利润估算、风险等级、竞争对手分析或投标建议。

### 2.3 协作与安全硬约束

- 多窗口只能修改各自声明的文件范围；未声明、范围重叠或归属不清时必须先协调。
- 每完成一个阶段，都必须更新该任务自己的独立日志。
- 每个任务使用一个独立日志文件；不允许多个窗口同时修改同一个日志。
- 不允许把账号、Cookie、Token、API Key 或其他凭据写入仓库、日志、示例、测试夹具或截图。
- 凭据只能通过仓库外的安全存储或运行时环境变量注入；日志只能记录变量名和配置状态，不能记录值。

## 3. 产品边界

### 3.1 当前用户与核心场景

当前产品主要服务投标方（乙方）：用户用自然语言描述关注主题、区域、时间和频率，系统采集招投标信息，整理甲方客观要求，识别新项目或实质变化，并交付 DOCX 报告。

### 3.2 当前阶段的产品输入与输出

输入：

- 用户的主题、区域、时间范围和执行频率。
- 合法访问的真实公开招投标网页与附件。

输出：

- 去重后的项目清单及原始来源。
- 可追溯的客观采购要求与证据位置。
- 新项目和实质字段变化。
- 可下载的 DOCX 报告。

不属于当前输出：企业能力画像、甲方预算预测、供应商匹配、胜率预测和投标策略。

## 4. 现有实现

### 4.1 前端

- 首页提供北京时间、自然语言输入、18 条本地推荐和四字段确认弹窗。
- 主题、区域、时间、频率由前端关键词/正则模拟提取，字段可编辑，并通过 `sessionStorage` 恢复。
- 项目列表、八类要求总览和要求详情页面已存在；支持桌面与移动布局。
- 项目列表中的“下载成 Word”按钮当前明确禁用。
- 报告页目前只是报告、推送记忆和来源状态的占位页面。

### 4.2 后端与工作流

- FastAPI 暴露任务运行、运行列表、项目列表、项目详情和模块详情接口。
- LangGraph 串联 15 个节点，包括需求理解、来源选择、采集、标准化、相关性、去重、证据、核验、变化检测和报告生成。
- SQLite 保存任务、运行结果、项目快照和项目展示资料；当前通过启动时 SQL 建表，没有正式迁移体系。
- 来源路由、项目归并、字段变化检测、证据绑定和质量重试已有可运行的原型结构。
- `design-qa.md` 记录 2026-07-13 时前端构建、页面测试和 Python 测试通过；本次初始化未重新执行这些测试。

## 5. 明确未实现或不能按真实能力验收的内容

- 三个数据源适配器均返回 `example.local` 模拟公告，没有连接真实政府平台、企业采购门户或商业平台。
- 登录型来源只有模拟会话状态，没有真实登录状态机、Cookie 安全存储、失效检测或人工接管流程。
- 附件获取器只返回“模拟附件正文”；没有真实附件下载、PDF 版面解析、扫描件 OCR 或 Office 文档解析。
- 文档解析器对 HTML 仅做标签清理；`pdf_layout`、`ocr_layout` 等目前只是路由标签，不代表解析已实现。
- 相关性判断是关键词包含规则；Evidence RAG 是本地词法/字符串相似度模拟，没有真实 LLM、Embedding 或向量检索。
- 八类项目要求及多数“原文位置”由 `project_profiles.py` 固定模板生成，不是从真实公告或附件逐条抽取；“第 1 页（模拟证据）”不能作为比赛证据。
- 当前去重基于项目编号、标题相似度、采购人和正文相似度，尚未用真实数据集验证误合并、漏合并和公告生命周期区分。
- 当前变化检测只比较 `budget`、`deadline`、`purchaser`，项目 ID 又由单次运行中的顺序生成；尚不能证明跨运行稳定增量。
- 调度器只计算下一次时间，没有常驻调度、来源水位线持久化、失败重试和推送闭环。
- 报告只写结构化 JSON；没有 DOCX 渲染器、格式验收或可用下载入口。
- 没有正式数据库迁移、生产级可观测性、来源限流/合规策略和真实数据端到端测试。
- 开发日志明确记录：当前没有甲方侧工具，也没有载入乙方企业信息；两者均不阻塞第一目标。

## 6. 当前决策

- 先保留现有模拟工作流作为可替换骨架，不把它包装成真实能力。
- 下一阶段从一个合法、稳定、权威的公开来源做纵向闭环，再扩展来源数量。
- 任何新功能若不能直接推进“真实数据源 → 清洗去重 → 增量判断 → DOCX”，默认排在第一目标之后。
- 所有真实事实必须携带来源、采集时间和可复核证据；无法确认的字段显示未知，不得由固定模板补造。
