# BidRadar-X AI 调用链路

## 一句话说明

AI 负责“理解、规划、复核和组织”，固定程序负责“登录授权、访问网站、保存原文、校验字段、控制费用和生成 Word”。没有 API Key 或模型调用失败时，系统自动回到原有规则链路，不要求用户确认，也不会中断任务。

## 完整链路

1. 用户输入自然语言问题。
2. 后端先用规则得到一个可靠的基础任务，再让 AI 输出结构化需求：主题、地区、关键词、排除词、时间范围和置信度。用户在表单中明确填写的主题、地区始终优先。
3. 需求理解完成后进入独立扩词节点。系统先用本地采购词典补充稳定同义词，再由智谱补充同义词、上下位品类词、设备形态、采购场景词和排除词。例如“服务器”会扩展到“机架式服务器、刀片服务器、GPU服务器、计算节点、存储服务器、超融合一体机”等表达。
4. AI 根据扩词结果、结构化需求和系统已登记的数据源生成检索式与来源建议。AI 只做计划，不直接自由浏览网页。
5. 固定来源适配器并行访问政府网站、公开平台和已授权 API。登录、Token、费用与超时均由程序控制。
6. 程序把各站结果转换为统一公告结构，保留标题、发布时间、原文链接、核心内容、附件链接和指纹。
7. 规则先做硬筛选，AI 再做语义相关性复核。AI 只能引用候选公告中真实出现的词，不能凭空扩展项目。
8. 稳定指纹先去重；只有标题、采购人或项目编号高度接近的歧义项目对才交给 AI。AI 给出高置信结论后，程序仍会再检查编号、标题相似度和采购人，满足安全阈值才合并。
9. 系统建立证据编号，把每个摘要和事实绑定到原公告内容与来源链接。
10. AI 做事实核验，发现没有证据或与证据冲突的结论时，系统自动再检索一次，不弹出用户确认。
11. AI 只生成带证据编号的执行摘要、关键发现、风险提示和建议动作。
12. 固定 Word 模板把 AI 摘要与原始公告字段组合成 DOCX。标题、发布时间、来源链接、核心原文和附件不由 AI 改写。
13. 系统重新打开生成的 DOCX，检查表格、链接和项目内容是否完整，随后保存报告和运行记录。

## Prompt 使用方式

不是每次手工改一大段 Prompt。系统保存六份版本化模板，只向模板注入本次任务的 JSON 数据：

- `intent-extraction@1.0.0`：需求理解
- `query-expansion@1.0.0`：采购检索扩词
- `search-planning@1.0.0`：检索规划
- `relevance-review@1.0.0`：相关性复核
- `ambiguous-deduplication@1.0.0`：歧义去重
- `fact-verification@1.0.0`：事实核验
- `evidence-report@1.0.0`：报告摘要

每次调用都要求模型按固定 JSON Schema 返回。返回内容如果不是合法 JSON、字段缺失、证据编号不存在、超时或限流，该步立即使用规则结果，不会把不合格内容写入报告。

## 密钥安全

- 前端代码没有 API Key，也没有读取 API Key 的能力。
- 不使用任何 `NEXT_PUBLIC_*` 环境变量保存模型密钥。
- 正式运行时推荐通过服务器或托管平台的 Secret 环境变量 `BIDRADAR_AI_API_KEY` 注入。
- 本地演示也可以由后端主机调用 `PUT /api/ai/credential`，密钥只保存在后端进程内存，重启即清除。该接口拒绝非本机访问，前端页面没有填写入口。
- 数据库只记录模型名、Prompt 版本、输入哈希、耗时、Token 用量、响应 ID 和成功/失败状态，不记录 API Key。

## 启用方式

后端检测到 `BIDRADAR_AI_API_KEY` 后自动启用，不需要再改代码，也不需要用户逐步确认。默认提供商是智谱，使用 `glm-5.2` 和 `https://open.bigmodel.cn/api/paas/v4/chat/completions`。模型、接口地址和超时可分别通过 `BIDRADAR_AI_MODEL`、`BIDRADAR_AI_BASE_URL`、`BIDRADAR_AI_TIMEOUT_SECONDS` 在后端调整。

`GET /api/ai/status` 只返回是否启用、模型名和接入阶段，不返回密钥。
