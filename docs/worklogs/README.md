# 工作日志协作规范

更新时间：2026-07-14（Asia/Shanghai）

## 核心规则

1. 每个任务必须使用一个独立日志文件，不允许多个任务共用一个日志。
2. 不允许多个窗口同时修改同一个日志文件。日志归任务窗口独占，直到任务移交或关闭。
3. 多窗口只能修改各自声明的文件范围。开始工作前，必须在独立日志中写明任务 ID、负责人/窗口和声明的文件范围。
4. 文件范围不得与其他进行中任务重叠。发现冲突时先停止修改，由协调窗口重新划分；不得以“最后合并”为理由并行改同一文件。
5. 每完成一个阶段都要更新自己的独立日志，至少记录完成内容、改动文件、验证结果、阻塞和下一步。
6. 不允许把账号、Cookie、Token、API Key 或任何凭据写入仓库或日志。只允许记录环境变量名、是否已配置和不含敏感值的错误摘要。

## 文件命名

复制 `TEMPLATE.md` 创建新日志，文件名使用：

```text
<任务ID>-<简短英文或拼音名称>.md
```

示例：

```text
R01-public-source.md
I02-change-rules.md
W02-docx-generator.md
```

不要把日志命名为 `current.md`、`work.md` 或按窗口编号命名；任务 ID 才是稳定归属。

## 认领流程

1. 从 `docs/WORK_PLAN.md` 选择依赖已满足的任务。
2. 确认没有同 ID 日志处于“进行中”，也没有其他任务声明重叠文件。
3. 从模板创建该任务的独立日志，填写负责人/窗口、状态和声明文件范围。
4. 先提交协调声明，再开始修改已声明文件。
5. 若必须扩大范围，先更新日志并确认没有冲突；不得先修改后补声明。

## 阶段更新要求

以下任一事件都视为一个阶段结束，必须立即更新日志：

- 完成设计或接口契约。
- 完成一组可独立验证的实现。
- 完成测试或验收。
- 遇到阻塞并停止推进。
- 准备移交给另一个窗口。
- 任务完成。

日志采用追加式阶段记录。可以修正文案，但不得删除失败、回滚、阻塞或验证不通过的历史。

## 移交与冲突处理

- 移交时，原窗口先在日志写明当前状态、已改文件、未完成项和下一位负责人；新窗口确认接管后才可继续。
- 一个日志在任一时刻只有一个写入者。协调窗口可以读取所有日志，但不得代替任务窗口写入。
- 若两个窗口误改同一业务文件，立即停止；保留双方工作副本，由协调窗口决定边界和整合方案，禁止互相覆盖。
- 若两个窗口误改同一日志，以最先完成有效认领的窗口为唯一写入者，另一个窗口新建自己的任务日志。

## 完成检查

任务日志标记“已完成”前必须同时满足：

- 验收条件逐条有结果。
- 改动文件清单完整。
- 验证命令和结果可复核。
- 未提交任何账号、Cookie、Token、API Key。
- 未修改声明范围外文件。
- `docs/WORK_PLAN.md` 的任务状态由协调窗口统一更新，任务窗口不得越权修改。
